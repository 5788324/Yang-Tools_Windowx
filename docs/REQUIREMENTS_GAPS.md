# 需求补充清单

这份清单记录 Yang Tools 在正式开发前需要补齐的产品、工程和协作要求。它会随着开发持续更新。

## 已明确需求

- 产品名：`Yang Tools`，属于 Yang 的个人 Windows 桌面工具箱。
- 参考：PowerToys、ZTools、uTools、PixPin。
- 重点插件：OCR、翻译、截图、钉图。
- 插件可安装、卸载、启用、禁用、更新。
- 支持 Codex、Claude Code、Antigravity 接力开发。
- 每天必须有工作日志和交接日志。
- 每次开始工作先拉取仓库，每次结束工作提交并推送。
- 本机 ZTools/uTools 插件先作为样本库，后续逐步适配。

## 建议补充的产品需求

- 明确主快捷键，建议默认 `Alt+Space`，同时允许用户改成 `Alt+Z` 或其他组合。
- 明确插件权限弹窗：安装插件时展示它需要剪贴板、文件、网络、截图等能力。
- 明确卸载策略：卸载插件时默认保留插件数据，提供“同时删除数据”选项。
- 明确更新策略：插件更新前备份旧版本，失败自动回滚。
- 明确离线模式：无网络时核心功能、已安装插件、本地搜索必须可用。
- 明确隐私策略：OCR/翻译如果调用联网服务，必须显示服务来源和数据会离开本机。
- 明确插件来源：本地 zip、GitHub Releases 插件索引、内置插件三种来源。

## 建议补充的插件兼容需求

- ZTools/uTools 插件分三类处理：
  - 直接兼容：manifest 简单、权限少。
  - 兼容桥运行：依赖 `window.ztools` / `window.utools` 常用 API。
  - 参考重写：依赖复杂、授权不清、风险高。
- `.asar` 插件只做本机只读分析，不提交源码。
- 第三方插件不默认进入发布包，除非确认许可证允许。

## 建议补充的工程需求

- 插件 API 必须有 schema 和类型定义。
- IPC 通道集中管理，避免插件绕过权限。
- 插件窗口和主窗口分开 preload。
- 插件数据按插件 ID 隔离。
- 日志文件不记录 API Key、剪贴板全文、OCR 原图等敏感数据。
- 后续增加自动化测试：manifest 解析、触发匹配、权限检查、安装/卸载/更新。

## 建议补充的交接要求

- `docs/HANDOFF.md` 永远记录最新状态。
- `docs/worklogs/YYYY-MM-DD.md` 每天一份。
- 长任务结束时必须写清楚下一步命令。
- 如果遇到上下文压缩，优先更新 `HANDOFF.md`。
- 如果 Claude Code/Antigravity 修改了代码，下一位 AI 必须先 `git pull` 并阅读最近 commit。

## 当前决策

- 第一阶段不追求完整复刻 uTools/ZTools。
- 第一阶段先实现 Yang Tools 自己的稳定插件协议和样本索引读取。
- ZTools/uTools 兼容作为运行时能力逐步加，不牺牲安全边界。
